## Problem Statement

Website owners, SEO specialists, and developers need to understand and control how AI crawlers interact with their content. The landscape of AI-related web standards is fragmented: robots.txt (legacy), llms.txt (new, Sept 2024 spec by Jeremy Howard), ai.txt (emerging), sitemaps (standard but under-leveraged for AI). No single free tool covers all of them in one place.

Existing tools each solve one slice:
- aicrawlercheck.com — robots.txt + bot directory (free, good but narrow)
- llmrefs.com — AI crawlability + llms.txt generator (paid, SEO-focused)
- knownagents.com — enterprise analytics, bot detection, auto robots.txt (paid, complex)
- llmstxt.org — the spec itself, not a tool

Nobody has built the **free, comprehensive, educational hub** that lets you check everything, generate everything, and actually understand what each file does.

## What Makes This Cool

**"AI Web Standards Hub"** — not just a checker tool, but THE reference site for understanding how websites talk to AI.

The "whoa" factor:
1. **Paste a URL, see everything** — one scan shows your robots.txt, llms.txt, ai.txt, sitemap status with an AI Readiness Score (0-100)
2. **Live playgrounds** — edit and preview llms.txt/robots.txt in real-time, see how different AI crawlers interpret your config
3. **Shareable reports** — `llmstxt.codes/report/example.com` with visual dashboard, sendable to dev team
4. **Education hub** — each file type has its own page with spec explanation, best practices, interactive examples, and AI crawler directory
5. **Generator tools** — guided wizards that create correct files based on your CMS/framework

## Constraints

### AI Readiness Score Formula

Each sub-score is 0-100:
- **File missing** → sub-score = 0
- **File exists but parse fails** → sub-score = 30 (partial credit for having the file)
- **File exists, valid format** → sub-score = 60-100 based on completeness

```
Score = (
  llms_txt_score * 0.30 +      -- Highest weight: llms.txt is the newest, most impactful standard
  robots_txt_score * 0.25 +    -- Has robots.txt? AI-specific rules? Correct syntax?
  sitemap_score * 0.15 +       -- Sitemap exists? Accessible? AI-optimized?
  ai_txt_score * 0.10 +        -- Has ai.txt? Valid format?
  meta_tags_score * 0.10 +     -- Correct meta robots tags? noai directives?
  headers_score * 0.10         -- X-Robots-Tag headers? AI-specific headers?
)

Letter grades: A+ (95-100), A (90-94), B (80-89), C (70-79), D (60-69), F (<60)
```

Weight rationale: llms.txt gets highest weight because it's the most actionable standard for AI readiness and the primary focus of this tool. robots.txt gets second because it's the most widely deployed. ai.txt/meta/headers are supplementary.

### Caching & Rate Limiting

- **Scan cache:** Results cached per domain for 24 hours in D1. Cache hit returns in <200ms.
- **Upstream fetch:** All 4 files (robots.txt, llms.txt, ai.txt, sitemap.xml) fetched **in parallel** using `Promise.allSettled()` with 3s timeout each. Worst-case scan time = 3s (parallel), not 12s (serial). Cloudflare Workers free tier allows 6 concurrent subrequests — 4 is within limit.
- **Rate limiting:** 10 scans/minute per IP (Cloudflare Workers built-in rate limiting)
- **Batch checker (v1.1):** 100 domains per request, 5 requests/hour per IP

### Error Handling

- **Domain unreachable:** Show "Could not connect to {domain}" with HTTP error code. Score stored as -1 (sentinel value), displayed as "N/A" in UI.
- **File not found (404):** Sub-score = 0, show "Not found — here's how to create one" with link to generator.
- **Parse error:** Sub-score = 30, show raw content with highlighted parse issues.
- **Timeout (>3s per file):** Sub-score = 0, show "Timeout — server took too long to respond."
- **Invalid UTF-8:** Attempt decoding as ISO-8859-1 fallback. If still invalid, sub-score = 10.
- **robots.txt parsing:** Use battle-tested `robots-parser` npm library, not custom parser. Handles wildcards, multiple User-Agent blocks, Allow/Disallow precedence correctly.

### Data Retention

- Scans retained in D1 for 90 days
- After 90 days, archived to R2 as compressed JSON
- Report URLs remain valid via R2 fallback (slower but functional)
- D1 capacity: ~1M records. Archive cron runs daily via Workers Cron Trigger
- **D1 free-tier limits:** 500MB storage, 5M reads/day, 100K writes/day. If approaching limits: (1) increase archive frequency, (2) show read-only cached results, (3) upgrade to D1 paid tier ($5/mo). Monitor via `scan_metrics` table.

### Architecture Change: Client-Side Parsing

**Critical finding:** Cloudflare Workers FREE tier = 10ms CPU per request. Parsing 4 files + scoring + D1 write = 60-220ms CPU. Impossible on free tier.

**Decision (auto, P5 explicit + P3 pragmatic):** Move parsing to client-side for MVP.

```
NEW ARCHITECTURE (MVP):
  Browser                         Worker (fetch proxy)         D1
  ──────                          ────────────────────         ──
  paste URL ──────────────────▶  /api/v1/fetch?domain=X
                                  fetch robots.txt  ──▶ target site
                                  fetch llms.txt    ──▶ target site
                                  fetch ai.txt      ──▶ target site
                                  fetch sitemap.xml ──▶ target site
  ◀────────────────── JSON ◀──  return { raw files }
  
  Parse robots.txt (robots-parser)
  Parse llms.txt (custom)
  Parse ai.txt (custom)
  Parse sitemap (check existence)
  Compute score
  ──────────────────────────────▶  /api/v1/save (scores + results_r2_key)
                                  write to D1 ─────────────────▶ D1
```

Worker CPU: <2ms (just fetch orchestration). All parsing = browser. $0 forever.

### Security Requirements Added
1. SSRF: domain validation (reject IPs, localhost, private ranges, .workers.dev). Use `psl` for domain extraction. `fetch(url, { redirect: 'manual' })`.

### Taste Decision: Thin Read-Only Score API in MVP

**The question:** Should MVP include `GET /api/v1/score/{domain}` — a read-only endpoint that returns cached scan results as JSON?

**What it enables:** CI/CD quality gates, badge SVGs, monitoring dashboards, programmatic access — all from a single endpoint reading D1 cache (<2ms Worker CPU).

**Arguments FOR (subagent):**
- ~4 hours of work for massive DX improvement (4/10 → ~6/10)
- Reads from D1 cache, not parsing — fits within Workers free tier CPU limits
- Unlocks the #1 developer use case (CI/CD integration)
- Badge SVG endpoint trivially derived from it
- Without it, scores are "vanity metrics" — can't fail a build

**Arguments AGAINST (auto-analysis):**
- Adds API surface to maintain from day 1
- Undocumented internal APIs attract scrapers (they'll use it even without docs)
- Rate limiting becomes more critical with programmatic access
- Scope creep: "just one endpoint" becomes "where's the POST? where's the webhook?"
- MVP focus is web UI for SEO-driven traffic, not API consumers

**NOT auto-decided.** This is a close call between "keep MVP focused on web" (P3 pragmatic) and "one endpoint unlocks the entire developer value prop" (P1 completeness, P4 user-centric). Surfaced to user at Final Approval Gate.
